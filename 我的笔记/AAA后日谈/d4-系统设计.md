# 系统设计

根据之前的需求分析，进行相应的系统设计：

## 用户端系统设计

1. 用户要能够通过**电脑的网页端**或**手机的小程序端**访问到医院的网站，同时，用户在不登录的状态下为游客模式，只能看到部分的公开信息。
2. 用户要能够看到最新发布的三条公告，以及所有待领养的宠物信息，和现在正在医院寄养的宠物信息。
3. 用户要能够看到当前医院的医生信息，包括他们的工号、照片、年龄、简介，以及医生当前的挂号占用情况。同时也能够直接向医生发起咨询或者求助。
4. 用户要能够查看自己的个人信息并对其进行修改，同时用户要能够查看自己的宠物信息，以及自己名下的挂号单详情。
5. 用户可以重置密码，或者在未登录的情况下进行注册，为了确保隐私以及数据安全，这些操作都要通过邮箱发送验证码的方式进行校验。

## 医生端系统设计

医生的设计大致与用户相同。

1. 要能够对自己当前挂号单进行处理，以向医院的前台发送反馈。
2. 要能够回答用户发起的在线咨询，这里的前端设计以及后台逻辑与用户端相同。
3. 能够修改自己部分的个人资料，以及通过邮箱发验证码的形式重置密码。

## 管理员端系统设计

1. 管理员要能够对用户、宠物、挂号、医生、领养、寄养、公告、科室、管理人员等信息进行增删改查。
2. 管理员要有权限的限制，比如医生、管理人员、科室这些信息只能由超级管理员进行修改和删除，普通管理员只能进行新增。同时，普通管理员在除了以上的三种信息外，与超级管理员具有相同的权限。而普通管理员之下还有普通员工，只能对用户、宠物、挂号、领养寄养这些信息进行新增。
3. 管理员在新增**部分涉及到多表操作的信息**时，应该有一个即时搜索功能。比如进行挂号操作，输入用户名时应该弹出模糊匹配的选项，比如输入“李”，会弹出“李四”、“李耳”的选项，选择用户名之后，下一个宠物名称输入栏应该只弹出该用户名下的宠物。
4. 同时，由于管理员端的在不同网页间的操作过于相似（都是增删改查），而所涉及的网页过多。故应当在实现的过程中，尽可能的抽离重复代码，封装为公用组件。
